PVector: Motion, Direction, and Steering in Processing

Why are we learning PVector?

Once motion gets even a little bit interesting, keeping track of separate variables like x, y, xspeed, yspeed, xacceleration, and yacceleration becomes messy. The Processing tutorial introduces PVector as a way to bundle related values together and work with motion more cleanly. Instead of treating horizontal and vertical motion as unrelated numbers, we can treat them as one object: a vector.

Main idea: a vector can represent position, velocity, acceleration, direction, force, or the difference between two points.

What a PVector stores

At its core, a PVector stores components such as x and y (and optionally z in 3D). That means a single variable can describe a position like (100, 200), a velocity like (2, -1), or an acceleration like (0, 0.1). 

PVector location = new PVector(100, 200);
PVector velocity = new PVector(2, -1);
PVector acceleration = new PVector(0, 0.1);

The Processing tutorial starts with the classic bouncing ball and rewrites it using PVector so that the motion rule becomes conceptually simple: 

location.add(velocity);

That line means: “move the object by its current velocity.” This is the heart of animation with vectors.

The motion chain

As our sketches become more realistic, we usually update motion in this order:

  1. Change velocity by acceleration
  2. Optionally limit the speed
  3. Change location by velocity
velocity.add(acceleration);
velocity.limit(topspeed);
location.add(velocity);
Conceptual summary:
position tells us where an object is,
velocity tells us how position changes,
acceleration tells us how velocity changes.

Useful PVector operations

Method What it does Typical use
add() Adds another vector Move position by velocity
sub() Subtracts one vector from another Find direction from one object to another
mult() Scales a vector Make a force stronger or weaker
div() Divides by a scalar Average values, normalize manually
mag() Returns magnitude (length) Measure distance-like size of a vector
normalize() Makes the vector have length 1 Keep only direction
limit() Caps magnitude Prevent runaway speed
heading() Returns direction as an angle Rotate a triangle/ship/boid to face its motion
PVector.angleBetween() Returns the smaller angle between two vectors Compare directions

Example 1: gravity and acceleration

Your gravity example is a good bridge from simple motion to more physical motion. The object has location, velocity, and acceleration, and gravity is modeled as a constant vertical acceleration. 

class Mover {
  PVector location;
  PVector velocity;
  PVector acceleration;
  float topspeed;

  Mover() {
    location = new PVector(width/2, 0);
    velocity = new PVector(1, 0);
    acceleration = new PVector(0, 0.01);
    topspeed = 6;
  }

  void update() {
    velocity.add(acceleration);
    velocity.limit(topspeed);
    location.add(velocity);
  }
}

This shows an important modeling idea: we do not usually set position directly once the sketch is running. Instead, we let acceleration influence velocity, and velocity influence location.

Notice: in the bouncing version of your code, the line velocity.y *= -0.95; makes the ball lose some energy after each bounce. That damping factor is a nice touch because it looks more realistic than a perfect bounce.

Example 2: moving with acceleration and screen wrap

Your second mover example starts in the center and uses a tiny constant acceleration. Instead of bouncing, it wraps around the screen. This is a nice chance to compare two boundary behaviors: bounce versus wrap. 

velocity.add(acceleration);
velocity.limit(topspeed);
location.add(velocity);
if (location.x > width) {
  location.x = 0;
} else if (location.x < 0) {
  location.x = width;
}

if (location.y > height) {
  location.y = 0;
} else if (location.y < 0) {
  location.y = height;
}

Wrapping is especially useful later for boids and particle systems, where objects leaving one side of the screen and reappearing on the other can make the world feel continuous.

Example 3: tip-to-tail vector addition

In your tip-to-tail demo, students click to create v₁ and v₂, then see the sum vector v₁ + v₂. This is exactly the geometric meaning of addition: copy the second vector so its tail starts at the tip of the first. 

PVector sum = PVector.add(v1, v2);
text("v₁ + v₂", sum.x + 10, sum.y);

The parallelogram fill is pedagogically useful because it shows that algebraic addition and geometric addition are describing the same object.

Good discussion question: Why does the green vector always go from the origin to the opposite corner of the parallelogram?

Example 4: vector subtraction

Subtraction is just as important as addition. In animation, subtraction often answers the question: What vector points from here to there? 

PVector diff = PVector.sub(v1, v2);

In your subtraction demo, students can see that v₁ - v₂ is the same as v₁ + (-v₂). That idea will matter later when a boid needs to compute a steering direction toward a target or away from a neighbor.

Very common pattern: 
PVector dir = PVector.sub(target, location);

This creates a vector pointing from the object's current position to the target.

Example 5: magnitude and normalization

The Processing tutorial emphasizes that the magnitude of a vector is its length. Once we know the magnitude, we can normalize the vector and keep only its direction. 

float m = dir.mag();
dir.normalize();

This distinction is crucial:

  • dir by itself contains both direction and size.
  • After normalize(), it keeps the same direction but has length 1.
  • Then mult(someValue) lets us decide how strong that direction should be.
dir.normalize();
dir.mult(0.5);

This “normalize, then scale” pattern appears constantly in motion and steering code.

Example 6: fireworks and particles

Your fireworks sketch is an excellent example of vectors inside objects. Each particle stores its own position, velocity, and acceleration: 

class Particle {
  PVector pos;
  PVector vel;
  PVector acc = new PVector(0, 0.05);
}

The update rule is written component-by-component in your code: 

vel.x += acc.x;
vel.y += acc.y;

pos.x += vel.x;
pos.y += vel.y;

That works, but this is also a good place to show students how vectors can shorten and clean up the code: 

vel.add(acc);
pos.add(vel);

The particle system also motivates arrays or ArrayLists of objects and shows why vectors matter in larger projects.

Example 7: angle between vectors

Your angleBetween sketch introduces a subtle but important point: PVector.angleBetween(v1, v2) returns the smaller unsigned angle between the vectors. 

float theta = PVector.angleBetween(v1, v2);

That means the result is between 0 and PI. It tells us how far apart the vectors are, but not whether the turn is clockwise or counterclockwise.

Important distinction: angleBetween() measures separation, not direction of rotation.

Example 8: signed angle using dot and cross products

Your signed-angle sketch fixes the limitation of angleBetween(). Using the dot product and the 2D cross-product idea, it computes a signed angle in the range -PI to PI. 

float signedAngle(PVector v1, PVector v2) {
  float cross = v1.x * v2.y - v1.y * v2.x;
  float dot = v1.x * v2.x + v1.y * v2.y;
  return atan2(cross, dot);
}

This is advanced for CS1, but it is a beautiful example of vectors carrying geometric meaning. It is especially useful when an object must decide whether to turn left or right.

Static vs. non-static style

Students will see both of these styles:

location.add(velocity);      // non-static: changes location
PVector sum = PVector.add(v1, v2);  // static: creates/returns a result

This is worth pointing out explicitly. Non-static calls often modify the vector you call them on. Static calls are often used when you want a new vector result.

Where this is heading

The tutorial gradually moves toward steering behavior: computing a vector toward the mouse, normalizing it, scaling it, and using it as acceleration. That gives us the core recipe for autonomous motion: 

PVector target = new PVector(mouseX, mouseY);
PVector dir = PVector.sub(target, location);
dir.normalize();
dir.mult(0.5);
acceleration = dir;

This is the bridge from “things moving on the screen” to “things choosing how to move.”

Possible project: flocking

A boid is a simulated bird-like agent. In a flocking model, each boid follows only local rules, but the group as a whole appears coordinated. Your flocking notes identify the three classic rules:

  1. Separation – avoid crowding nearby boids
  2. Alignment – steer toward the average direction of nearby boids
  3. Cohesion – steer toward the average position of nearby boids

This is a natural next step after PVector because every boid needs a position, velocity, and acceleration vector, and the three rules are built out of subtraction, addition, averaging, normalization, and limiting.

Suggested flocking project stages

Stage Goal Vector ideas involved
1 Create one moving boid object location, velocity, acceleration, update(), display()
2 Rotate the boid so it faces its direction of motion velocity.heading()
3 Add many boids with an array or ArrayList Objects plus repeated vector updates
4 Implement screen wrap Continuous world idea
5 Implement separation Subtract nearby positions, steer away
6 Implement alignment Average neighbors' velocities
7 Implement cohesion Average neighbors' positions, then steer toward that center
8 Tune the weights of the three rules mult(), limit()

Starter structure for a boid

class Boid {
  PVector location;
  PVector velocity;
  PVector acceleration;
  float maxSpeed;
  float maxForce;

  Boid(float x, float y) {
    location = new PVector(x, y);
    velocity = PVector.random2D();
    acceleration = new PVector(0, 0);
    maxSpeed = 3;
    maxForce = 0.08;
  }

  void update() {
    velocity.add(acceleration);
    velocity.limit(maxSpeed);
    location.add(velocity);
    acceleration.mult(0);
  }

  void applyForce(PVector force) {
    acceleration.add(force);
  }

  void edges() {
    if (location.x < 0) location.x = width;
    if (location.x > width) location.x = 0;
    if (location.y < 0) location.y = height;
    if (location.y > height) location.y = 0;
  }

  void display() {
    float theta = velocity.heading();
    pushMatrix();
    translate(location.x, location.y);
    rotate(theta);
    triangle(12, 0, -10, 6, -10, -6);
    popMatrix();
  }
}

This is not the full flocking logic yet. It is the shell students can build on once they understand vectors.

Takeaways

  • PVector helps us organize motion-related quantities cleanly.
  • Most motion code follows the pattern: acceleration → velocity → location.
  • Normalization separates direction from size.
  • Subtraction is the key to “where should I go?”
  • Limiting magnitude helps control speed and steering strength.
  • These ideas lead naturally to particles, steering, and flocking.
